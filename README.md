# IBN Workflow — Intent Dataset Generation

A pipeline for generating high-quality, platform-aware datasets to train LLMs (LLaMA-8B, Phi, Gemma) to convert natural-language network intents into structured JSON.

---

## Overview

The dataset is built through five stages:

1. Device list creation
2. Intent dataset generation
3. Data profiling
4. Data validation
5. JSONL conversion for parameter-efficient fine-tuning

**Final outputs:**
- `intent_dataset.json` — raw structured dataset
- `llama_training.jsonl` — LLaMA-ready supervised fine-tuning dataset

---

## Components

### 1. Intent Templates

**File:** `intent_template.json`

Defines natural-language patterns for each intent type and sub-intent type.

```json
"vlan": {
  "create": {
    "allow": [
      "{action} vlan {vlan_id} on {device} port {port}",
      "{action} vlan {vlan_id} tagged on {device} port {port}"
    ]
  }
}
```

---

### 2. Policy Templates

**File:** `policy_template.json`  
**Generated by:** `policygenerator.py`

Defines required/optional parameters, constraints, and allowed actions per intent.

```json
"vlan": {
  "create": {
    "allow": {
      "required_parameters": ["device", "port", "vlan_id"],
      "optional_parameters": ["tagging_mode"]
    }
  }
}
```

---

### 3. Platform Capabilities

Each platform supports only specific intent types, ensuring the dataset is platform-correct.

```json
"raspberrypi": [
  "system_mgmt",
  "vlan",
  "lldp",
  "snmp",
  "monitoring"
]
```

---

### 4. Platform Parameter Constraints

Each platform enforces its own hardware-level constraints:

| Constraint | Description |
|---|---|
| `port_range` | Valid port identifiers for the platform |
| `mtu_range` | Minimum and maximum MTU values |
| `qos_supported` | Whether QoS is available |
| `poe_supported` | Whether Power over Ethernet is available |
| `lldp_med_supported` | Whether LLDP-MED is available |

Example:

```json
"CLASSIC": {
  "port_range": ["1/1", "1/2", "1/3", "1/8"],
  "mtu_range": [576, 1500],
  "qos_supported": false,
  "poe_supported": false
}
```

---

### 5. Device Inventory

**File:** `devices_list.json`  
**Generated by:** `createdevicelist.py`, `ports.py`

Defines device names, platforms, IP addresses, and available ports.

```json
"HIOS": {
  "brs40-1": {
    "ip": "10.0.0.1",
    "ports": ["1/1", "1/2", "1/3"]
  }
}
```

---

### 6. Dataset Generator

**Script:** `generatedata.py`  
**Run:** `python datafortraining/generatedata.py`  
**Output:** `datafortraining/intent_dataset.json`

**Features:**
- Platform-aware intent selection
- Platform-aware parameter constraints
- Noise-injected natural language for robustness
- Required/optional parameter enforcement
- Template placeholder filling
- Policy-aligned parameter filtering

**Sample output record:**

```json
{
  "uid": "uuid",
  "natural_language": "configure vlan 10 tagged on brs40-1 port 1/1",
  "intent_type": "vlan",
  "intent_sub_type": "create",
  "intent_action": "allow",
  "parameters": {
    "device": "brs40-1",
    "port": "1/1",
    "vlan_id": 10,
    "tagging_mode": "tagged"
  }
}
```

---

### 7. Dataset Verification and Profiling

Three scripts handle quality checks:

| Script | Purpose | Output |
|---|---|---|
| `datavalidation.py` | Validates required parameters, platform constraints, port ranges, and unsupported actions | `validation_report.txt` |
| `checknull.py` | Detects and reports null or empty values | `empty_values_report.txt` |
| `dataprofiling.py` | Reports missing values, parameter distribution, intent type frequency, and data spread | `data_profiling_report.txt` |

---

### 8. Pre-Training Dataset Test (Ollama)

Before fine-tuning, the dataset is tested against a base model (`phi3:mini`) via [Ollama](https://ollama.com) using `basemodelcheck.py` to verify:

- Natural-language clarity
- JSON formatting consistency
- Instruction consistency

> **Note:** This is a quality and clarity check, not a schema correctness test.

## Fine-Tuning

### Parameter-Efficient Fine-Tuning (PEFT)

Rather than full fine-tuning (expensive and slow), this project uses PEFT methods which are faster, cheaper, more stable, and possible on CPU or small GPU.

Two methods are used:

#### LoRA — Low-Rank Adapters (16-bit precision)

- Injects small trainable matrices into the model's attention layers
- Only 0.1–1% of model parameters are updated
- Good balance of quality and speed

#### QLoRA — Quantized Low-Rank Adapters (4-bit precision)

- Loads the base model in 4-bit quantization
- Trains LoRA adapters on top of the quantized model
- Most memory-efficient option — suitable for consumer hardware

---

### Model Selection

**LLaMA** is the primary model for this project due to its strengths in:

- Understanding noisy natural language
- Following structured templates
- Intent classification tasks
